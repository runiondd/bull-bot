# CoveredCallOverlay Strategy — Design Spec

**Date:** 2026-04-14
**Status:** Approved

## Overview

A new strategy class that sells short-dated calls against existing long call (LEAPS) positions and shares. Unlike other strategies, CoveredCallOverlay does not open the primary position — it reads a `long_inventory` table of positions the user already holds and manages only the short call overlay. The evolver optimizes parameters through backtesting like any other strategy.

## Motivation

Dan holds 10 TSLA LEAPS contracts plus shares across a taxable brokerage and a tax-sheltered IRA. The LEAPS are the growth exposure; selling short-dated calls against them generates premium income. This income is especially important in the taxable brokerage account as a cash flow source.

## Architecture

```
long_inventory (DB table)          CoveredCallOverlay strategy
  - your existing LEAPS    ──>     - reads inventory
  - shares                         - sells short calls against them
  - cost basis                     - manages rolls and closes
  - account type                   - evolver optimizes params
```

The strategy fits into the existing Bull-Bot framework:
- Registered in the strategy registry like any other strategy
- Evaluated by `engine.step()` on each bar
- Proposals generated by the evolver/proposer
- Backtested via walk-forward harness
- Promoted through paper trial

## New Database Table: `long_inventory`

Stores the user's existing long positions that short calls can be sold against.

```sql
CREATE TABLE long_inventory (
    id INTEGER PRIMARY KEY,
    account TEXT NOT NULL,        -- 'taxable' or 'ira'
    ticker TEXT NOT NULL,
    kind TEXT NOT NULL,           -- 'call' or 'shares'
    strike REAL,                  -- NULL for shares
    expiry TEXT,                  -- 'YYYY-MM-DD', NULL for shares
    quantity REAL NOT NULL,       -- contracts for calls, share count for shares
    cost_basis_per REAL,          -- per-contract or per-share cost basis
    added_at INTEGER NOT NULL,
    removed_at INTEGER            -- NULL while active
);
```

Seeded from the Fidelity CSV export. Updated manually when the user trades LEAPS. The strategy reads rows where `removed_at IS NULL`.

## Strategy Class: `CoveredCallOverlay`

### Evolver Parameters

**Strike selection:**
- `short_delta`: float (0.10–0.50) — target delta for the short call. Higher = more premium, more assignment risk.

**Timing:**
- `dte_min`: int — minimum DTE for the short call (e.g., 14)
- `dte_max`: int — maximum DTE for the short call (e.g., 60)
- `coverage_ratio`: float (0.0–1.0) — fraction of long contracts to sell against. 1.0 = fully covered, 0.67 = leave some uncapped.

**Entry timing filters:**
- `min_rsi`: float — only sell when RSI is above this. Prevents selling into oversold conditions where a bounce is likely. (e.g., 45)
- `min_day_return`: float — only sell on days when the stock is up at least this much. Sell into strength, not weakness. (e.g., 0.01 = 1%)
- `iv_rank_min`: float — prefer selling when IV is elevated for fatter premium. (e.g., 0)

**Roll/close management:**
- `roll_dte`: int — roll the short call when it reaches this many DTE (e.g., 5)
- `profit_target_pct`: float — close the short call at this % of max profit (e.g., 0.50)
- `roll_itm_delta`: float — if short call delta exceeds this (gone ITM), trigger defensive roll (e.g., 0.70)
- `defend_time_value_min`: float — minimum time value remaining on the long call (in dollars) to justify rolling up/out vs letting assignment happen. If the long call has little time value left, it's not worth defending.

### evaluate() Logic

On each bar:

1. **Load inventory**: Query `long_inventory` for active long positions for this ticker.

2. **Load existing short calls**: Check `positions` table for open CoveredCallOverlay positions for this ticker/run_id.

3. **Check exits on existing short calls**:
   - If a short call hits `profit_target_pct`, close it.
   - If a short call reaches `roll_dte`, roll it (close + open new one).
   - If a short call's delta exceeds `roll_itm_delta` (ITM threat), evaluate whether to roll or let assign based on `defend_time_value_min` of the underlying long call.

4. **Check for new short call opportunities**:
   - Count uncovered long contracts (inventory minus existing short calls).
   - Apply `coverage_ratio` to determine how many short calls to sell.
   - Check timing filters: RSI >= `min_rsi`, day return >= `min_day_return`, IV rank >= `iv_rank_min`.
   - If all filters pass, generate a signal to sell short calls at the target delta/DTE.

### Signal Structure

The signal intent is "open" with a single short call leg. The `max_loss_per_contract` is technically unlimited for a naked short call, but since it's covered by the long position, the real risk is the spread between long and short strikes. For shares, max loss is capped by the share cost basis minus the short strike.

### Account-Aware Behavior

The evolver can discover different parameter sets per account type. The proposer guidance should note that taxable positions prioritize premium income (higher delta, more frequent selling) while IRA positions can optimize for total return (lower delta, less frequent).

The strategy includes `account` as a filter — when evaluating, it only considers inventory matching the configured account. Two separate strategy instances can be proposed: one for taxable, one for IRA.

## Walk-Forward Considerations

During backtesting, the strategy needs synthetic long inventory to sell against. The walk-forward harness should:
- Use the `long_inventory` table as-is for paper/live trading
- For backtesting, assume a standing long call position existed throughout the backtest period (since we're testing the overlay logic, not the LEAPS entry)

The backtest evaluates: given that you held a LEAPS position, how much premium could the overlay have generated, and what was the drawdown when short calls went ITM?

### Backtest Inventory Synthesis

For backtesting, create a synthetic long inventory entry: one ATM LEAPS call per ticker, refreshed annually. This gives the overlay something to sell against during the walk-forward without requiring the actual historical LEAPS position. The evolver evaluates the overlay's premium generation and risk management, not the LEAPS entry timing.

## Integration Points

- **Strategy registry**: Register `CoveredCallOverlay` with `CLASS_NAME = "CoveredCallOverlay"`.
- **Proposer guidance**: Add PMCC-specific guidance for growth tickers that have `long_inventory` entries.
- **Position sizer**: Short calls are sized by the inventory (contracts available to cover), not by the risk fraction. The `coverage_ratio` parameter controls how many to sell.
- **Fill model**: Uses existing `simulate_open_multi_leg` for selling the short call. Single leg (short call only).
- **Exit manager**: Uses existing profit target and DTE-based exits. Add ITM roll logic.
- **Dashboard**: Show CoveredCallOverlay positions with their associated long inventory context — which LEAPS they're sold against, premium collected to date, net cost basis reduction.

## Out of Scope

- Automatic LEAPS entry (that's GrowthLEAPS, separate strategy)
- Fidelity API integration (manual CSV import for now)
- Non-TSLA positions (design supports any ticker, but TSLA is the priority)
- Diagonal spreads with different expiry buckets (short call is always shorter-dated than the long)
